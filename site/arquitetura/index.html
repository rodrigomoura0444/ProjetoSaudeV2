    ProjetoSaudeV2 — System Architecture

    1. Overview

    ProjetoSaudeV2 is structured around four core layers that communicate through HTTP requests and local data storage mechanisms.

    [ Python Simulator ]
        │  POST /paciente (JSON)
        ▼
[ Node-RED ] → Processing & Validation → [ SQLite Database ]
     │
     └──► uibuilder (HTML/CSS/JS) → Web Dashboard (Real-Time)
    • Python — Generates and transmits simulated patient vital signs.
    • Node-RED — Receives, validates, and stores the data.
    • SQLite — Maintains the historical record of all measurements.
    • uibuilder — Provides a responsive and interactive web dashboard for real-time visualization.

    2. Components and Roles

    2.1 Python (simular_enviar.py)

    • Generates randomized vital sign data (heart rate, temperature, SpO₂, blood pressure).
    • Sends this data to Node-RED via an HTTP POST request to /paciente.

    2.2 Node-RED

    • HTTP In /paciente — Receives JSON input from the Python simulator.
    • Function: Format external batch — Validates and normalizes incoming data.
    • Function: Build INSERT in vitals — Constructs SQL statements for database storage.
    • SQLite (INSERT vitals) — Saves all validated data.
    • HTTP GET /api/v1/vitals — Exposes historical data through an API endpoint.
    • uibuilder — Pushes live updates to the frontend dashboard.

    2.3 SQLite

    • Stores patient records and corresponding vital measurements.
    • Provides fast and lightweight local data access, avoiding remote server dependencies.

    2.4 uibuilder

    • Displays real-time patient information in a web-based interface.
    • Implements a responsive layout using standard HTML, CSS, and JavaScript technologies.

    3. Data Flow

    A) Data Ingestion (Python → Node-RED → SQLite → UI)

    1. Python sends data batches to Node-RED via POST /paciente.
    2. Node-RED processes and stores the data in SQLite.
    3. uibuilder reflects the updated information automatically on the dashboard.

    B) Data Query (UI → API)

    1. The user interface or external client sends a request to /api/v1/vitals?patient_id=1&minutes=60.
    2. Node-RED queries SQLite and returns a structured JSON response.

    4. API Endpoints

    POST /paciente

    Request Example:

    [
  {
    "paciente": "Maria Silva",
    "batimentos": 78,
    "temperatura": 36.7,
    "spo2": 97,
    "pa": { "sistolica": 120, "diastolica": 80 },
    "ts": 1739600000000
  }
]

    Response:
    HTTP 200 OK

    GET /api/v1/vitals?patient_id={id}&minutes={window}

    Response Example:

    {
  "data": [
    {
      "ts_ms": 1739600000000,
      "heart_rate": 78,
      "temperature": 36.7,
      "spo2": 97,
      "systolic": 120,
      "diastolic": 80
    }
  ]
}

    5. Database Model (SQLite)

    CREATE TABLE patients (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  birthdate TEXT,

);

CREATE TABLE vitals (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  patient_id INTEGER NOT NULL REFERENCES patients(id) ON DELETE CASCADE,
  ts_ms INTEGER NOT NULL,
  heart_rate INTEGER,
  temperature REAL,
  spo2 INTEGER,
  systolic INTEGER,
  diastolic INTEGER
);

CREATE INDEX idx_vitals_patient_ts ON vitals(patient_id, ts_ms DESC);

    6. Node-RED Flow Design

    [HTTP in POST /paciente]
     ↓
[Function: Format external batch]
     ↓
[Function: Build INSERT in vitals]
     ↓
[SQLite: INSERT vitals]
     ↓
[HTTP response 200]

[HTTP in GET /api/v1/vitals]
     ↓
[Function: Build query (OK/ERROR)]
   ├── OK → [SQLite: SELECT history] → [Function: to JSON] → [HTTP 200]
   └── ERROR → [HTTP error]

    7. Summary

    The architecture of ProjetoSaudeV2 is modular, scalable, and maintainable.
    It combines Python-based simulation, Node-RED automation, SQLite persistence, and uibuilder visualization into a cohesive ecosystem for real-time patient monitoring.
